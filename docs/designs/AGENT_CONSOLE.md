## Problem Statement

AI agents can now do real, substantive work — write code, run tests, draft documents, deploy infrastructure. But every tool built to watch and manage them assumes the person watching is a developer. Code editors, Superset, observability platforms — all of them require reading diffs, terminal output, or stack traces to know what's happening.

The person actually responsible for directing an agent — deciding what it does next, approving the risky calls, watching what it's costing — is increasingly not a developer at all. As agent fleets scale past one or two agents into a real roster, that gap stops being an inconvenience and becomes a liability: the manager either has to become a developer or fly blind and trust.

The gap is not a UI skin problem. It is a primitives problem: every existing tool speaks in the language of code (traces, spans, diffs, token logs). The non-developer director needs a tool that speaks in the language of decisions (what should this agent do next? is this safe to approve? what did we spend?).

**A second dimension of the same problem — and the more urgent one:** An agent that hits a judgment call it cannot make alone does not just pause on a risky action. It blocks entirely, writes a note to a mailbox file in the control repo, and waits. The human must discover this has happened (by checking the mailbox file in git), read the task context (by finding and reading the task file in git), make a decision (in a separate Claude conversation, copying task context in by hand), and then write the unblock command back into the control repo from a terminal. The loop is: git → terminal → new Claude tab → git → terminal. Every step requires developer tooling. Until now, there has been no way to close this loop from a browser.

## Demand Evidence

- **Active internal use case:** cstack fleet running real ECOBA work today. The problem is being felt, not hypothesized. The CLI supervisor (supervisor/watch.ts) is already in use; its limitation — requiring a terminal, readable only by developers — is a real constraint on who can direct the fleet.
- **Landscape confirmation:** Every existing AI observability tool (Langfuse, AgentOps, Arize, LangSmith, Kore.ai) targets technical users. The 2026 AI governance literature recognizes that non-technical managers need to be in approval loops but provides no UX for it — only API webhooks.
- **Zero competition in this specific seat:** No existing tool is designed for the non-developer who is accountable for what agents do and what they cost. The gap is real and unserved.

## Status Quo

The person directing the cstack fleet today must:
1. Open a terminal and run `bun supervisor/watch.ts` to see agent status
2. Interpret ANSI-colored terminal output (agent state, task ID, last tool name)
3. Infer whether an action is risky from raw tool names (e.g., "Bash (git push)")
4. Check cost by running `python3 supervisor/analytics.py` separately
5. Approve/reject actions by... having no mechanism — agents run autonomously until they hit a cost guard
6. Discover a blocked task by periodically checking `mailboxes/tshepo.md` in the control repo — an agent that needs a decision writes a short note there and parks itself with `status: needs_human`
7. Navigate to the task file (`tasks/<TASK-ID>.md`) in the control repo to read the full context
8. Open a new Claude conversation, copy the task context in by hand, and make a decision together with Claude
9. Return to the terminal and run `kernel/task unblock <id> --agent tshepo --reason "<decision>"` to flip the task back to `status: open` so the agent can pick it up again

Steps 6–9 are the live workflow today. They require a terminal, a working knowledge of the control repo directory structure, and the ability to correctly invoke the kernel CLI. Nothing surfaces the blocked task to the director automatically — if you forget to check the mailbox file, the task sits parked indefinitely.

The supervisor infrastructure already captures the right data (live.json, live-events.jsonl, METRICS.jsonl). The task ledger captures blocked state. The problem is translation and routing: the data exists in a form that requires a developer to find and interpret it, and the write-back (unblocking a task) requires a terminal command.

## Target User & Narrowest Wedge

**Target user (immediate):** The person running the cstack/ECOBA engagement — directing 4 agents (agent-be, agent-fe, agent-qa, agent-doc), responsible for output quality, cost, and risk approval. Currently requires terminal access and developer literacy to do this job.

**Narrowest wedge:** The **approval queue** — a mechanism for agents to surface "I want to do X, should I?" and for the director to approve or reject in plain English, without a terminal. This is the one thing that does not exist yet in the CLI supervisor. Everything else (status dashboard, cost bar) is data that already exists rendered in a browser. The approval queue is new infrastructure and new UX.

**What a non-developer can now do with this:** Open a browser tab, see what the fleet is doing, approve a git push or reject an API call, read what a blocked agent needs, make a decision, and send the agent back to work — all without a terminal or git.

**Two distinct interaction types the console must handle:**
- **Approval Queue** — agent wants to execute a risky action right now. Binary decision, synchronous: approve or reject before the agent continues.
- **Human Attention Queue** — agent hit a judgment call it cannot make alone. The task is parked (`status: needs_human`). The director reads context, makes a decision, and unblocks the task when ready — asynchronous, the agent is already parked and waiting.

## Constraints

- **macOS-first:** The fleet runs on macOS; the console must work on the same machine. Multi-machine / remote access is out of scope for v1.
- **Minimal agent-side changes:** The console reads what stream-processor.py, run-agent.sh, and analytics.py already write. run-agent.sh gains ~20 lines for risk gating and approval polling (see Approval Protocol). stream-processor.py gains one new event type. No new infrastructure, no new processes.
- **Ship fast:** Intrapreneurship goal. One ECOBA session to working prototype. No build step, no framework churn.
- **Single user for v1:** No auth, no multi-user. It's a local browser tab.

## Approaches Considered

### Approach A: Browser Tab on Existing CLI Data (Minimal Viable) ✓ CHOSEN
- **Summary:** Local Bun + Hono HTTP server reads live.json / live-events.jsonl / METRICS.jsonl and serves an htmx + SSE browser UI. Triage-first: approval queue is the home screen, status strip below it.
- **Effort:** S (human: ~1-2 days / CC: ~1 hour)
- **Risk:** Low
- **Pros:** Zero new infrastructure. Ships today. No build step, no framework. Validates UX premise before investing in full architecture.
- **Cons:** Single-machine only. Approval write-back needs a new polling file in run-agent.sh.
- **Reuses:** live.json, live-events.jsonl, METRICS.jsonl, stream-processor.py, analytics.py

### Approach B: Full Web App + WebSocket + Persistence
- **Summary:** React/SvelteKit frontend + Bun/Node backend, WebSocket real-time, SQLite audit log, agent SDK approval queue write-back, two-tier chat, setup flow, optional auth.
- **Effort:** L (human: 1-2 weeks / CC: ~2-3 hours)
- **Risk:** Medium
- **Pros:** Right long-term architecture. Supports multi-user, remote access, audit trail.
- **Cons:** Risk of over-building before validating UX. 2-3x more moving parts.

### Approach C: Electron Desktop App (macOS Native)
- **Summary:** Electron + React, reads local files directly, macOS notifications for pending approvals, Dock badge for queue depth.
- **Effort:** M (human: 3-5 days / CC: ~1.5 hours)
- **Risk:** Medium
- **Pros:** Native macOS feel. Dock badge is lowest-friction "something needs you" signal. Distributable .app.
- **Cons:** macOS-only. Electron is a large dependency. Harder to generalize to web/cloud.

## Recommended Approach

**Approach A: Browser Tab on Existing CLI Data**

Rationale: The intrapreneurship goal is validate fast, not build the platform. The approval queue is the one new primitive that doesn't exist yet; everything else is data the CLI supervisor already produces. Ship the browser tab, use it for one full ECOBA session, then decide whether B or C is the right evolution based on what you learn.

### Architecture (Approach A)

```
supervisor/console/
├── server.ts          # Bun + Hono HTTP server, port 7842
├── index.html         # htmx-driven, no build step
├── styles.css         # Dark theme, design tokens from DESIGN.md
└── decisions/         # Approval write-back: run-agent.sh polls this dir
    └── {agent-name}-{request_id}.json   # one file per pending request
```

All paths are relative to the repo root. run-agent.sh references decisions as `$(git rev-parse --show-toplevel)/supervisor/console/decisions/`.

The server needs one additional env var at startup: `CONTROL_DIR` — the path to the control repo (where `tasks/`, `mailboxes/`, and `kernel/` live). This is already set in the supervisor environment. If absent, the Human Attention Queue section renders empty with an error note.

**CONTROL_DIR auto-detection (D2.3):** If `CONTROL_DIR` is not set, server.ts reads `supervisor/fleet.conf` to find agent names, then checks if `~/agents/<first-agent>/config` exists and reads `CONTROL_REPO_URL` from it. It then clones a dedicated console checkout: `git clone $CONTROL_REPO_URL ~/agents/console/control`. On subsequent starts, the checkout is pulled to latest. The console always uses `~/agents/console/control` as its CONTROL_DIR — it NEVER shares a checkout with any running agent. This eliminates `.git/index.lock` races between the console's git operations and agents' concurrent git operations.

**Port binding:** Server binds to `127.0.0.1:7842`, not `0.0.0.0`. This is enforced in Hono: `app.listen({ port: 7842, hostname: '127.0.0.1' })`. On a shared or cloud workstation, binding to 0.0.0.0 would expose unauthenticated RCE surface (POST /api/unblock runs kernel commands). Localhost-only is the correct default for v0.

**Real-time event push (SSE):** The approval flow is synchronous — the agent is blocked waiting for up to 10 minutes. A 5s poll delay on approval card appearance is unacceptable UX. The server watches live-events.jsonl via `fs.watch()` and pushes new `approval_requested` and `needs_human` events to all connected clients via SSE immediately. The 5s poll (`/api/queue`) remains as a reconnection fallback (SSE reconnects automatically; the poll catches any events missed during reconnection).

**Decisions file schema (Approval Queue):**
```json
{
  "request_id": "uuid-v4",
  "agent": "agent-fe",
  "action": "git push origin main",
  "description": "Wants to push 3 commits to origin/main. Changes include updated auth flow and a dependency bump.",
  "risk": "high",
  "files": ["src/auth/session.ts", "src/middleware/auth.ts", "package.json"],
  "created_at": "2026-06-19T11:42:00Z",
  "approved": null,
  "decided_at": null
}
```

After the user clicks Approve or Reject, the console server updates `approved` (true/false) and `decided_at` in-place by overwriting the file atomically (write to `{file}.tmp`, then `rename`). run-agent.sh detects the update by polling `approved !== null`.

**Data flow:**
```
live.json ──────────────────────────────────────────── SSE /stream/status
live-events.jsonl ──────────────────────────────────── SSE /stream/events
METRICS.jsonl ────── analytics.py ──► /api/cost ─────── cost bar

# Approval Queue (synchronous — agent is paused waiting)
run-agent.sh ──► writes decisions/{agent}-{id}.json ─── renders in approval queue
user clicks Approve/Reject ──► POST /api/approve/:id ──► server overwrites file ──► run-agent.sh resumes

# Human Attention Queue (asynchronous — agent is already parked)
$CONTROL_DIR/tasks/*.md (status: needs_human) ──────── renders in attention queue
$CONTROL_DIR/mailboxes/tshepo.md ────────────────────── agent's reason shown in card
user types decision + clicks Unblock ──► POST /api/unblock/:taskId ──► kernel/task unblock ──► task → open → agent picks up
```

**SSE reconnect:** htmx's `hx-ext="sse"` reconnects automatically on connection drop. The server re-streams from the tail of live-events.jsonl on each new SSE connection, so no state is lost on server restart.

### Approval Protocol (new, minimal)

**Risk interception mechanism — Bash wrapper on PATH:**

Claude CLI executes tools locally (runs bash, returns result to the API). There is no hook point between Claude's tool_use decision and local execution — stream-processor.py only sees tool_call events after the tool has already dispatched. Intercepting at the stream level is too late for prevention.

The correct interception point is the tool itself. Before starting Claude, run-agent.sh prepends `$SUPERVISOR_DIR/console/bin/` to PATH. This directory contains a script `bash` (the risk wrapper) that intercepts every Bash tool call at execution time:

```bash
# supervisor/console/bin/bash — risk wrapper (replaces real bash on PATH for Claude sessions)
#!/usr/bin/env bash
set -u
REAL_BASH="$(type -ap bash | grep -v "$0" | head -1)"
COMMAND="$*"

check_risk() {
  local cmd="$1"
  # HIGH: git push, git reset --hard, rm -rf outside repo, curl to non-localhost
  echo "$cmd" | grep -qE '^(git push|git reset --hard|git force-push|curl http[^/]|rm -rf /)'  && { echo "high"; return; }
  # MEDIUM: git commit, bun add, npm install, sudo
  echo "$cmd" | grep -qE '^(git commit|bun add|npm install|sudo)' && { echo "medium"; return; }
  echo "low"
}

RISK=$(check_risk "$COMMAND")
if [ "$RISK" = "high" ] || [ "$RISK" = "medium" ]; then
  REQUEST_ID=$(uuidgen | tr '[:upper:]' '[:lower:]' | cut -c1-8)
  DECISIONS_FILE="$SUPERVISOR_DECISIONS_DIR/${AGENT_NAME}-${REQUEST_ID}.json"
  printf '{"request_id":"%s","agent":"%s","action":"%s","description":"%s","risk":"%s","files":[],"created_at":"%s","approved":null,"decided_at":null}\n' \
    "$REQUEST_ID" "$AGENT_NAME" "$COMMAND" "Wants to run: $COMMAND" "$RISK" "$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
    > "${DECISIONS_FILE}.tmp" && mv "${DECISIONS_FILE}.tmp" "$DECISIONS_FILE"
  printf '{"ts":"%s","agent":"%s","type":"approval_requested","request_id":"%s","tool":"Bash","summary":"%s","risk":"%s","files":[]}\n' \
    "$(date -u +%Y-%m-%dT%H:%M:%SZ)" "$AGENT_NAME" "$REQUEST_ID" "$COMMAND" "$RISK" \
    >> "$EVENTS_FILE"
  RESULT=$(poll_approval "$REQUEST_ID" "$DECISIONS_FILE")
  if [ "$RESULT" != "true" ]; then
    echo "Action rejected or timed out. Aborting: $COMMAND" >&2
    exit 1
  fi
fi
exec "$REAL_BASH" "$@"
```

**Required env vars in the bash wrapper (set by run-agent.sh before starting Claude):**
- `AGENT_NAME` — agent identifier (already set by run-agent.sh)
- `SUPERVISOR_DECISIONS_DIR` — path to `supervisor/console/decisions/`
- `EVENTS_FILE` — path to `$LOG_DIR/live-events.jsonl`

**run-agent.sh changes (~20 lines):**
1. Export `SUPERVISOR_DECISIONS_DIR` and `EVENTS_FILE`
2. Prepend `$SUPERVISOR_DIR/console/bin/` to PATH before launching Claude subprocess
3. The `poll_approval` function (named pipe poll, see below) is sourced from a shared lib or inlined

The console server watches live-events.jsonl for `type == "approval_requested"` to populate the queue. It does NOT scan the decisions/ directory — the event log is the source of truth for the queue.

**`files` field convention:** When the action has no relevant files (e.g., an external API call), set `"files": []`. The console renders this as "no files affected."

**Blocking concern — non-blocking poll (macOS-safe):** `timeout` (GNU coreutils) is not available on stock macOS. Using a sleep-loop-with-countdown instead. The poll runs as a background subshell writing to a named pipe; a cleanup trap handles crashes:

```bash
# run-agent.sh: approval poll (macOS-safe, non-blocking via named pipe)
FIFO=$(mktemp -u /tmp/gstack-approval-XXXXXX)
mkfifo "$FIFO"
trap "kill \$POLL_PID 2>/dev/null; rm -f '$FIFO'" EXIT INT TERM
(
  DEADLINE=$(( $(date +%s) + 600 ))  # 10 minutes
  while [ "$(date +%s)" -lt "$DEADLINE" ]; do
    sleep 2
    APPROVED=$(jq -r '.approved // empty' "$DECISIONS_FILE" 2>/dev/null)
    if [ -n "$APPROVED" ]; then
      printf '%s\n' "$APPROVED" > "$FIFO"
      exit 0
    fi
  done
  printf 'timeout\n' > "$FIFO"
) &
POLL_PID=$!
RESULT=$(cat "$FIFO")
wait $POLL_PID 2>/dev/null
trap - EXIT INT TERM; rm -f "$FIFO"
```

Tmp file convention: `{agent-name}-{request_id}.json.tmp` (e.g., `agent-fe-550e8400.json.tmp`), in the same `decisions/` directory. After writing, `mv decisions/{agent}-{id}.json.tmp decisions/{agent}-{id}.json` (atomic on macOS, same filesystem).

**run-agent.sh changes (~30 lines total, concentrated in one function):**
1. `check_risk(tool, command)` → returns HIGH/MEDIUM/LOW
2. `request_approval(agent, tool, command, risk, files)` → writes decisions file, appends `approval_requested` event to live-events.jsonl, returns request_id
3. `poll_approval(request_id, risk_level)` → macOS-safe poll via named pipe, returns "true"/"false"/"timeout"

**Timeout behavior:**
- LOW risk: auto-approve after 10 minutes (low-stakes, unblocking)
- MEDIUM risk: auto-reject after 10 minutes (conservative default)
- HIGH risk: auto-reject after 10 minutes (conservative default)
All timeouts emit an `approval_timeout` event to live-events.jsonl.

**`approval_requested` event schema** (new event type for stream-processor.py / live-events.jsonl):
```json
{
  "ts": "2026-06-19T11:42:00Z",
  "agent": "agent-fe",
  "type": "approval_requested",
  "request_id": "uuid-v4",
  "tool": "Bash",
  "summary": "git push origin main",
  "risk": "high",
  "files": ["src/auth/session.ts", "package.json"]
}
```
The console server watches live-events.jsonl for `type == "approval_requested"` events and renders them as cards in the approval queue. This avoids the console having to scan the decisions/ directory.

**Risk classification (initial, hardcoded in run-agent.sh):**
- HIGH: `git push`, `git reset --hard`, `git force-push`, external API calls (curl/fetch to non-localhost), file deletes outside working dir (`rm -rf`), database migrations
- MEDIUM: `git commit`, `npm install`/`bun add`, schema migrations, `sudo` commands
- LOW: reads, writes inside repo, test runs (`bun test`, `pytest`), linting

### Human Attention Queue Protocol

**What triggers it:** An agent that cannot make a judgment call alone runs `kernel/task fail <id> --agent $AGENT_NAME --role $AGENT_ROLE --needs-human`. This:
1. Sets `status: needs_human` in the task ledger (`ledger/<id>.task`)
2. Increments `failure_count` on the task
3. The agent is then required (by AGENT_BASE rules) to append a short entry to `$CONTROL_DIR/mailboxes/tshepo.md` explaining what decision is needed and pointing at the task ID

The task sits parked indefinitely. Only `kernel/task unblock <id>` (human-only command) moves it back to `status: open`.

**Data sources the console reads:**
- `$CONTROL_DIR/ledger/*.task` — the kernel ledger files. Filter for `status: needs_human`. Each ledger file contains `task_id`, `status`, `agent`, `spec`, `failure_count`, `updated_at`.
- `$CONTROL_DIR/tasks/<TASK-ID>.md` — the task spec file. Contains the task title, acceptance criteria, context, and any Q&A the agent wrote explaining the blocker.
- `$CONTROL_DIR/mailboxes/tshepo.md` — the human mailbox. The agent writes a short "re: <task-id>" block explaining what it needs. The console parses this to show the agent's note on each blocked task card.

**How the console reads blocked tasks (`GET /api/attention`):**
```typescript
// server.ts
app.get('/api/attention', async (c) => {
  const controlDir = process.env.CONTROL_DIR;
  if (!controlDir) return c.json({ items: [], error: 'CONTROL_DIR not set' });

  // Read ledger files for needs_human status
  const ledgerDir = path.join(controlDir, 'ledger');
  const blockedTasks = [];
  for (const file of await readdir(ledgerDir)) {
    const content = await readFile(path.join(ledgerDir, file), 'utf-8');
    const entry = parseTaskLedger(content);  // key:value frontmatter parser
    if (entry.status === 'needs_human') {
      const specPath = path.join(controlDir, 'tasks', `${entry.task_id}.md`);
      const spec = await readFile(specPath, 'utf-8').catch(() => '');
      blockedTasks.push({ ...entry, spec });
    }
  }

  // Parse mailbox for per-task agent notes
  const mailbox = await readFile(path.join(controlDir, 'mailboxes', 'tshepo.md'), 'utf-8').catch(() => '');
  const notes = parseMailboxNotes(mailbox);  // returns Map<taskId, string>

  return c.json({ items: blockedTasks.map(t => ({ ...t, agentNote: notes.get(t.task_id) ?? '' })) });
});
```

**How the console unblocks a task (`POST /api/unblock/:taskId`):**
```typescript
app.post('/api/unblock/:taskId', async (c) => {
  const { taskId } = c.req.param();
  const { reason, agentName } = await c.req.json();
  if (!reason?.trim()) return c.json({ error: 'reason required' }, 400);

  const controlDir = process.env.CONTROL_DIR;
  const kernelPath = path.join(controlDir, 'kernel', 'task');

  // Run kernel/task unblock — this is a git commit in the control repo
  const result = await Bun.spawn([kernelPath, 'unblock', taskId, '--agent', 'tshepo', '--reason', reason]);
  const ok = (await result.exited) === 0;

  if (ok && agentName) {
    const knownAgents = getKnownAgentsFromFleetConf();  // validated at startup
    if (knownAgents.has(agentName)) {
      const mailboxPath = path.join(controlDir, 'mailboxes', `${agentName}.md`);
      const msg = `\n## from: tshepo | ${new Date().toISOString()} | re: ${taskId}\n${reason}\n`;
      await appendFile(mailboxPath, msg);
      await gitCommitAndPush(controlDir, [`mailboxes/${agentName}.md`], `mailbox: tshepo → ${agentName} re: ${taskId}`);
    }
  }
  return c.json({ ok });
});
```

**agentName path traversal protection:** `agentName` from the URL is validated against the Set of known agents read from `fleet.conf` at server startup. Unknown names return 400. This prevents path traversal attacks through the mailbox write path.

**What "Send back to agent" does (end-to-end):**
1. Human reads the blocked task card in the console (task title, agent's note, relevant ACs from the spec)
2. Human types their decision in the textarea and clicks "Send back to agent →"
3. Console POSTs `{ reason, agentName }` to `/api/unblock/:taskId`
4. Server runs `kernel/task unblock <id> --agent tshepo --reason "<decision>"` — this commits to the control repo, flipping `status: needs_human → open`
5. Server appends the decision to `mailboxes/<agent-name>.md` — the agent will read this on its next session start and see the resolution
6. The task is now eligible for any agent with the matching domain to pick up (or the original agent, when it wakes and finds it in the eligible pool)
7. If the supervisor's wake system is running, the original agent receives a wake signal within 1–5 seconds and starts a new session picking up the now-open task

**Mailbox write-back is a UX enhancement, not required for correctness.** The kernel's `task unblock` alone is sufficient to make the task eligible. The mailbox write-back gives the agent the human's decision as context when it picks up the task, so it doesn't have to re-derive what to do.

**Git push retry for mailbox write:** On push rejection, `git pull --rebase` then push again. One retry is sufficient for a 4-agent fleet:

```typescript
async function gitCommitAndPush(controlDir: string, files: string[], message: string) {
  const run = (args: string[]) => Bun.spawn(['git', '-C', controlDir, ...args]);
  await run(['add', ...files]).exited;
  await run(['commit', '-m', message]).exited;
  const push1 = await run(['push']).exited;
  if (push1 !== 0) {
    await run(['pull', '--rebase']).exited;
    await run(['push']).exited;  // second failure propagates to caller
  }
}
```

**Task ledger format** (`.task` files are plain key:value, one pair per line):
```
task_id: FEAT-011
status: needs_human
agent: agent-fe
role: feature
spec: tasks/FEAT-011.md
failure_count: 1
blocked_by:
lease_expires:
updated_at: 2026-06-19T09:28:00Z
```

**The console does NOT write to the task spec file** (`tasks/<TASK-ID>.md`). The spec is the agent's artifact; the console only reads it to surface context to the human.

**Failure count context for the UI:** If `failure_count >= 2`, the task has been attempted and blocked more than once. Show amber warning: "This task has been blocked twice — the AC may need rewriting."

1. **What defines a "high risk" action?** The initial list above is a starting point. After one ECOBA session, audit which approvals felt unnecessary and which felt critical — then tune the classification.
2. **How does the director talk to one agent directly?** The mailbox write-back from `POST /api/unblock` handles task-scoped messages. General mailbox messaging (non-task-scoped) is deferred to v2.

5. **Should the Human Attention Queue surface `awaiting_info` tasks?** No. `awaiting_info` is the agent-to-agent Q&A mechanism. The console only surfaces `needs_human` (after agent-to-agent escalation failed or wasn't applicable).

### Design System Notes (corrections from DESIGN.md)

These correct silent mismatches between the original wireframe and DESIGN.md:

1. **Body font size:** `16px` (not 14px). DESIGN.md: `Body: 16px`.
2. **Button border-radius:** `8px` (not 5px). DESIGN.md: `Buttons/inputs: md (8px)`.
3. **Font loading:** Three `<link>` tags required — Satoshi from Fontshare, DM Sans + JetBrains Mono from Google Fonts with `display=swap`. Without these, browser falls back to system-ui (DESIGN.md prohibits default font stacks).
4. **Grain texture:** `body::after` noise overlay — `feTurbulence` SVG, `opacity: 0.03`, `position: fixed`, `pointer-events: none`, `z-index: 9999`. See DESIGN.md "Grain Texture" section.

The AI draft panel sits **between** the AC context block and the decision textarea in the Human Attention card. It is collapsible (hidden by default; toggle "AI Draft ↓" button reveals it).

Position: top-right of the card meta row (same row as agent name, task ID, blocked timer).

### Dynamic Browser Tab Title

```javascript

  document.title = pendingCount > 0
    ? `(${pendingCount}) Fleet Console`
    : 'Fleet Console — All clear';

```

Call on every htmx poll response. `pendingCount` = attention queue items + approval queue items combined.

### Keyboard Navigation & ARIA (minimal spec)

1. **Tab order:** page header → attention queue cards → [for each: textarea → Send button → AI Draft toggle] → approval queue cards → [for each: Reject button → Approve button]
2. **Enter on Approve button:** fires approval (same as click). No accidental triggering via form submit.
3. **Focus after approve/reject:** move focus to next card in queue, or to the "No pending approvals" empty state if queue is now empty.
4. **ARIA landmarks:** `<section aria-label="Human Attention Queue">`, `<section aria-label="Approval Queue">`, each card as `<article aria-label="[agent] — [task/action]">`.
5. **Live regions:** `aria-live="polite"` on waiting timers (update without interrupting screen reader), `role="alert"` on error cards, `role="status"` on success flash and empty states.
6. **Queue count in tab title** also serves as a screen-reader-accessible notification (document title changes are announced by assistive tech).
