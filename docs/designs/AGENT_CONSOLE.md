## Problem Statement

AI agents can now do real, substantive work — write code, run tests, draft documents, deploy infrastructure. But every tool built to watch and manage them assumes the person watching is a developer. Code editors, Superset, observability platforms — all of them require reading diffs, terminal output, or stack traces to know what's happening.

The person actually responsible for directing an agent — deciding what it does next, approving the risky calls, watching what it's costing — is increasingly not a developer at all. As agent fleets scale past one or two agents into a real roster, that gap stops being an inconvenience and becomes a liability: the manager either has to become a developer or fly blind and trust.

The gap is not a UI skin problem. It is a primitives problem: every existing tool speaks in the language of code (traces, spans, diffs, token logs). The non-developer director needs a tool that speaks in the language of decisions (what should this agent do next? is this safe to approve? what did we spend?).

**A second dimension of the same problem — and the more urgent one:** An agent that hits a judgment call it cannot make alone does not just pause on a risky action. It blocks entirely, writes a note to a mailbox file in the control repo, and waits. The human must discover this has happened (by checking the mailbox file in git), read the task context (by finding and reading the task file in git), make a decision (in a separate Claude conversation, copying task context in by hand), and then write the unblock command back into the control repo from a terminal. The loop is: git → terminal → new Claude tab → git → terminal. Every step requires developer tooling. Until now, there has been no way to close this loop from a browser.

## Demand Evidence

- **Active internal use case:** cstack fleet running real ECOBA work today. The problem is being felt, not hypothesized. The CLI supervisor (supervisor/watch.ts) is already in use; its limitation — requiring a terminal, readable only by developers — is a real constraint on who can direct the fleet.
- **Landscape confirmation:** Every existing AI observability tool (Langfuse, AgentOps, Arize, LangSmith, Kore.ai) targets technical users. The 2026 AI governance literature recognizes that non-technical managers need to be in approval loops but provides no UX for it — only API webhooks.
- **Zero competition in this specific seat:** No existing tool is designed for the non-developer who is accountable for what agents do and what they cost. The gap is real and unserved.

## Status Quo

The person directing the cstack fleet today must:
1. Open a terminal and run `bun supervisor/watch.ts` to see agent status
2. Interpret ANSI-colored terminal output (agent state, task ID, last tool name)
3. Infer whether an action is risky from raw tool names (e.g., "Bash (git push)")
4. Check cost by running `python3 supervisor/analytics.py` separately
5. Approve/reject actions by... having no mechanism — agents run autonomously until they hit a cost guard
6. Discover a blocked task by periodically checking `mailboxes/tshepo.md` in the control repo — an agent that needs a decision writes a short note there and parks itself with `status: needs_human`
7. Navigate to the task file (`tasks/<TASK-ID>.md`) in the control repo to read the full context
8. Open a new Claude conversation, copy the task context in by hand, and make a decision together with Claude
9. Return to the terminal and run `kernel/task unblock <id> --agent tshepo --reason "<decision>"` to flip the task back to `status: open` so the agent can pick it up again

Steps 6–9 are the live workflow today. They require a terminal, a working knowledge of the control repo directory structure, and the ability to correctly invoke the kernel CLI. Nothing surfaces the blocked task to the director automatically — if you forget to check the mailbox file, the task sits parked indefinitely.

The supervisor infrastructure already captures the right data (live.json, live-events.jsonl, METRICS.jsonl). The task ledger captures blocked state. The problem is translation and routing: the data exists in a form that requires a developer to find and interpret it, and the write-back (unblocking a task) requires a terminal command.

## Target User & Narrowest Wedge

**Target user (immediate):** The person running the cstack/ECOBA engagement — directing 4 agents (agent-be, agent-fe, agent-qa, agent-doc), responsible for output quality, cost, and risk approval. Currently requires terminal access and developer literacy to do this job.

**Narrowest wedge:** The **approval queue** — a mechanism for agents to surface "I want to do X, should I?" and for the director to approve or reject in plain English, without a terminal. This is the one thing that does not exist yet in the CLI supervisor. Everything else (status dashboard, cost bar) is data that already exists rendered in a browser. The approval queue is new infrastructure and new UX.

**What a non-developer can now do with this:** Open a browser tab, see what the fleet is doing, approve a git push or reject an API call, read what a blocked agent needs, make a decision, and send the agent back to work — all without a terminal or git.

**Two distinct interaction types the console must handle:**
- **Approval Queue** — agent wants to execute a risky action right now. Binary decision, synchronous: approve or reject before the agent continues.
- **Human Attention Queue** — agent hit a judgment call it cannot make alone. The task is parked (`status: needs_human`). The director reads context, makes a decision, and unblocks the task when ready — asynchronous, the agent is already parked and waiting.

- **macOS-first:** The fleet runs on macOS; the console must work on the same machine. Multi-machine / remote access is out of scope for v1.
- **Minimal agent-side changes:** The console reads what stream-processor.py, run-agent.sh, and analytics.py already write. run-agent.sh gains ~20 lines for risk gating and approval polling (see Approval Protocol). stream-processor.py gains one new event type. No new infrastructure, no new processes.
- **Ship fast:** Intrapreneurship goal. One ECOBA session to working prototype. No build step, no framework churn.
- **Single user for v1:** No auth, no multi-user. It's a local browser tab.

## Approaches Considered

### Approach A: Browser Tab on Existing CLI Data (Minimal Viable) ✓ CHOSEN
- **Summary:** Local Bun + Hono HTTP server reads live.json / live-events.jsonl / METRICS.jsonl and serves an htmx + SSE browser UI. Triage-first: approval queue is the home screen, status strip below it.
- **Effort:** S (human: ~1-2 days / CC: ~1 hour)
- **Risk:** Low
- **Pros:** Zero new infrastructure. Ships today. No build step, no framework. Validates UX premise before investing in full architecture.
- **Cons:** Single-machine only. Approval write-back needs a new polling file in run-agent.sh.
- **Reuses:** live.json, live-events.jsonl, METRICS.jsonl, stream-processor.py, analytics.py

### Approach B: Full Web App + WebSocket + Persistence
- **Summary:** React/SvelteKit frontend + Bun/Node backend, WebSocket real-time, SQLite audit log, agent SDK approval queue write-back, two-tier chat, setup flow, optional auth.
- **Effort:** L (human: 1-2 weeks / CC: ~2-3 hours)
- **Risk:** Medium
- **Pros:** Right long-term architecture. Supports multi-user, remote access, audit trail.
- **Cons:** Risk of over-building before validating UX. 2-3x more moving parts.

### Approach C: Electron Desktop App (macOS Native)
- **Summary:** Electron + React, reads local files directly, macOS notifications for pending approvals, Dock badge for queue depth.
- **Effort:** M (human: 3-5 days / CC: ~1.5 hours)
- **Risk:** Medium
- **Pros:** Native macOS feel. Dock badge is lowest-friction "something needs you" signal. Distributable .app.
- **Cons:** macOS-only. Electron is a large dependency. Harder to generalize to web/cloud.

## Recommended Approach

**Approach A: Browser Tab on Existing CLI Data**

Rationale: The intrapreneurship goal is validate fast, not build the platform. The approval queue is the one new primitive that doesn't exist yet; everything else is data the CLI supervisor already produces. Ship the browser tab, use it for one full ECOBA session, then decide whether B or C is the right evolution based on what you learn.

### Architecture (Approach A)

```
supervisor/console/
├── server.ts          # Bun + Hono HTTP server, port 7842
├── index.html         # htmx-driven, no build step
├── styles.css         # Dark theme, design tokens from DESIGN.md
└── decisions/         # Approval write-back: run-agent.sh polls this dir
    └── {agent-name}-{request_id}.json   # one file per pending request
```

All paths are relative to the repo root. run-agent.sh references decisions as `$(git rev-parse --show-toplevel)/supervisor/console/decisions/`.

The server needs one additional env var at startup: `CONTROL_DIR` — the path to the control repo (where `tasks/`, `mailboxes/`, and `kernel/` live). This is already set in the supervisor environment. If absent, the Human Attention Queue section renders empty with an error note.

**CONTROL_DIR auto-detection (D2.3):** If `CONTROL_DIR` is not set, server.ts reads `supervisor/fleet.conf` to find agent names, then checks if `~/agents/<first-agent>/config` exists and reads `CONTROL_REPO_URL` from it. It then clones a dedicated console checkout: `git clone $CONTROL_REPO_URL ~/agents/console/control`. On subsequent starts, the checkout is pulled to latest. The console always uses `~/agents/console/control` as its CONTROL_DIR — it NEVER shares a checkout with any running agent. This eliminates `.git/index.lock` races between the console's git operations and agents' concurrent git operations.

**Port binding:** Server binds to `127.0.0.1:7842`, not `0.0.0.0`. This is enforced in Hono: `app.listen({ port: 7842, hostname: '127.0.0.1' })`. On a shared or cloud workstation, binding to 0.0.0.0 would expose unauthenticated RCE surface (POST /api/unblock runs kernel commands). Localhost-only is the correct default for v0.

**Real-time event push (SSE):** The approval flow is synchronous — the agent is blocked waiting for up to 10 minutes. A 5s poll delay on approval card appearance is unacceptable UX. The server watches live-events.jsonl via `fs.watch()` and pushes new `approval_requested` and `needs_human` events to all connected clients via SSE immediately. The 5s poll (`/api/queue`) remains as a reconnection fallback (SSE reconnects automatically; the poll catches any events missed during reconnection).

**Decisions file schema (Approval Queue):**
```json
{
  "request_id": "uuid-v4",
  "agent": "agent-fe",
  "action": "git push origin main",
  "description": "Wants to push 3 commits to origin/main. Changes include updated auth flow and a dependency bump.",
  "risk": "high",
  "files": ["src/auth/session.ts", "src/middleware/auth.ts", "package.json"],
  "created_at": "2026-06-19T11:42:00Z",
  "approved": null,
  "decided_at": null
}
```

After the user clicks Approve or Reject, the console server updates `approved` (true/false) and `decided_at` in-place by overwriting the file atomically (write to `{file}.tmp`, then `rename`). run-agent.sh detects the update by polling `approved !== null`.

**Data flow:**
```
live.json ──────────────────────────────────────────── SSE /stream/status
live-events.jsonl ──────────────────────────────────── SSE /stream/events
METRICS.jsonl ────── analytics.py ──► /api/cost ─────── cost bar

# Approval Queue (synchronous — agent is paused waiting)
run-agent.sh ──► writes decisions/{agent}-{id}.json ─── renders in approval queue
user clicks Approve/Reject ──► POST /api/approve/:id ──► server overwrites file ──► run-agent.sh resumes

# Human Attention Queue (asynchronous — agent is already parked)
$CONTROL_DIR/tasks/*.md (status: needs_human) ──────── renders in attention queue
$CONTROL_DIR/mailboxes/tshepo.md ────────────────────── agent's reason shown in card
user types decision + clicks Unblock ──► POST /api/unblock/:taskId ──► kernel/task unblock ──► task → open → agent picks up
```

**SSE reconnect:** htmx's `hx-ext="sse"` reconnects automatically on connection drop. The server re-streams from the tail of live-events.jsonl on each new SSE connection, so no state is lost on server restart.

### Approval Protocol (new, minimal)

**Risk interception mechanism — Bash wrapper on PATH:**

Claude CLI executes tools locally (runs bash, returns result to the API). There is no hook point between Claude's tool_use decision and local execution — stream-processor.py only sees tool_call events after the tool has already dispatched. Intercepting at the stream level is too late for prevention.

The correct interception point is the tool itself. Before starting Claude, run-agent.sh prepends `$SUPERVISOR_DIR/console/bin/` to PATH. This directory contains a script `bash` (the risk wrapper) that intercepts every Bash tool call at execution time:

```bash
# supervisor/console/bin/bash — risk wrapper (replaces real bash on PATH for Claude sessions)
#!/usr/bin/env bash
set -u
REAL_BASH="$(type -ap bash | grep -v "$0" | head -1)"
COMMAND="$*"

check_risk() {
  local cmd="$1"
  # HIGH: git push, git reset --hard, rm -rf outside repo, curl to non-localhost
  echo "$cmd" | grep -qE '^(git push|git reset --hard|git force-push|curl http[^/]|rm -rf /)'  && { echo "high"; return; }
  # MEDIUM: git commit, bun add, npm install, sudo
  echo "$cmd" | grep -qE '^(git commit|bun add|npm install|sudo)' && { echo "medium"; return; }
  echo "low"
}

RISK=$(check_risk "$COMMAND")
if [ "$RISK" = "high" ] || [ "$RISK" = "medium" ]; then
  REQUEST_ID=$(uuidgen | tr '[:upper:]' '[:lower:]' | cut -c1-8)
  DECISIONS_FILE="$SUPERVISOR_DECISIONS_DIR/${AGENT_NAME}-${REQUEST_ID}.json"
  printf '{"request_id":"%s","agent":"%s","action":"%s","description":"%s","risk":"%s","files":[],"created_at":"%s","approved":null,"decided_at":null}\n' \
    "$REQUEST_ID" "$AGENT_NAME" "$COMMAND" "Wants to run: $COMMAND" "$RISK" "$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
    > "${DECISIONS_FILE}.tmp" && mv "${DECISIONS_FILE}.tmp" "$DECISIONS_FILE"
  printf '{"ts":"%s","agent":"%s","type":"approval_requested","request_id":"%s","tool":"Bash","summary":"%s","risk":"%s","files":[]}\n' \
    "$(date -u +%Y-%m-%dT%H:%M:%SZ)" "$AGENT_NAME" "$REQUEST_ID" "$COMMAND" "$RISK" \
    >> "$EVENTS_FILE"
  RESULT=$(poll_approval "$REQUEST_ID" "$DECISIONS_FILE")
  if [ "$RESULT" != "true" ]; then
    echo "Action rejected or timed out. Aborting: $COMMAND" >&2
    exit 1
  fi
fi
exec "$REAL_BASH" "$@"
```

**Required env vars in the bash wrapper (set by run-agent.sh before starting Claude):**
- `AGENT_NAME` — agent identifier (already set by run-agent.sh)
- `SUPERVISOR_DECISIONS_DIR` — path to `supervisor/console/decisions/`
- `EVENTS_FILE` — path to `$LOG_DIR/live-events.jsonl`

**run-agent.sh changes (~20 lines):**
1. Export `SUPERVISOR_DECISIONS_DIR` and `EVENTS_FILE`
2. Prepend `$SUPERVISOR_DIR/console/bin/` to PATH before launching Claude subprocess
3. The `poll_approval` function (named pipe poll, see below) is sourced from a shared lib or inlined

The console server watches live-events.jsonl for `type == "approval_requested"` to populate the queue. It does NOT scan the decisions/ directory — the event log is the source of truth for the queue.

**`files` field convention:** When the action has no relevant files (e.g., an external API call), set `"files": []`. The console renders this as "no files affected."

**Blocking concern — non-blocking poll (macOS-safe):** `timeout` (GNU coreutils) is not available on stock macOS. Using a sleep-loop-with-countdown instead. The poll runs as a background subshell writing to a named pipe; a cleanup trap handles crashes:

```bash
# run-agent.sh: approval poll (macOS-safe, non-blocking via named pipe)
FIFO=$(mktemp -u /tmp/gstack-approval-XXXXXX)
mkfifo "$FIFO"
trap "kill \$POLL_PID 2>/dev/null; rm -f '$FIFO'" EXIT INT TERM
(
  DEADLINE=$(( $(date +%s) + 600 ))  # 10 minutes
  while [ "$(date +%s)" -lt "$DEADLINE" ]; do
    sleep 2
    APPROVED=$(jq -r '.approved // empty' "$DECISIONS_FILE" 2>/dev/null)
    if [ -n "$APPROVED" ]; then
      printf '%s\n' "$APPROVED" > "$FIFO"
      exit 0
    fi
  done
  printf 'timeout\n' > "$FIFO"
) &
POLL_PID=$!
RESULT=$(cat "$FIFO")
wait $POLL_PID 2>/dev/null
trap - EXIT INT TERM; rm -f "$FIFO"
```

Tmp file convention: `{agent-name}-{request_id}.json.tmp` (e.g., `agent-fe-550e8400.json.tmp`), in the same `decisions/` directory. After writing, `mv decisions/{agent}-{id}.json.tmp decisions/{agent}-{id}.json` (atomic on macOS, same filesystem).

**run-agent.sh changes (~30 lines total, concentrated in one function):**
1. `check_risk(tool, command)` → returns HIGH/MEDIUM/LOW
2. `request_approval(agent, tool, command, risk, files)` → writes decisions file, appends `approval_requested` event to live-events.jsonl, returns request_id
3. `poll_approval(request_id, risk_level)` → macOS-safe poll via named pipe, returns "true"/"false"/"timeout"

**Timeout behavior:**
- LOW risk: auto-approve after 10 minutes (low-stakes, unblocking)
- MEDIUM risk: auto-reject after 10 minutes (conservative default)
- HIGH risk: auto-reject after 10 minutes (conservative default)
All timeouts emit an `approval_timeout` event to live-events.jsonl.

**`approval_requested` event schema** (new event type for stream-processor.py / live-events.jsonl):
```json
{
  "ts": "2026-06-19T11:42:00Z",
  "agent": "agent-fe",
  "type": "approval_requested",
  "request_id": "uuid-v4",
  "tool": "Bash",
  "summary": "git push origin main",
  "risk": "high",
  "files": ["src/auth/session.ts", "package.json"]
}
```
The console server watches live-events.jsonl for `type == "approval_requested"` events and renders them as cards in the approval queue. This avoids the console having to scan the decisions/ directory.

**Risk classification (initial, hardcoded in run-agent.sh):**
- HIGH: `git push`, `git reset --hard`, `git force-push`, external API calls (curl/fetch to non-localhost), file deletes outside working dir (`rm -rf`), database migrations
- MEDIUM: `git commit`, `npm install`/`bun add`, schema migrations, `sudo` commands
- LOW: reads, writes inside repo, test runs (`bun test`, `pytest`), linting

### Human Attention Queue Protocol

**What triggers it:** An agent that cannot make a judgment call alone runs `kernel/task fail <id> --agent $AGENT_NAME --role $AGENT_ROLE --needs-human`. This:
1. Sets `status: needs_human` in the task ledger (`ledger/<id>.task`)
2. Increments `failure_count` on the task
3. The agent is then required (by AGENT_BASE rules) to append a short entry to `$CONTROL_DIR/mailboxes/tshepo.md` explaining what decision is needed and pointing at the task ID

The task sits parked indefinitely. Only `kernel/task unblock <id>` (human-only command) moves it back to `status: open`.

**Data sources the console reads:**
- `$CONTROL_DIR/ledger/*.task` — the kernel ledger files. Filter for `status: needs_human`. Each ledger file contains `task_id`, `status`, `agent`, `spec`, `failure_count`, `updated_at`.
- `$CONTROL_DIR/tasks/<TASK-ID>.md` — the task spec file. Contains the task title, acceptance criteria, context, and any Q&A the agent wrote explaining the blocker.
- `$CONTROL_DIR/mailboxes/tshepo.md` — the human mailbox. The agent writes a short "re: <task-id>" block explaining what it needs. The console parses this to show the agent's note on each blocked task card.

**How the console reads blocked tasks (`GET /api/attention`):**
```typescript
// server.ts
app.get('/api/attention', async (c) => {
  const controlDir = process.env.CONTROL_DIR;
  if (!controlDir) return c.json({ items: [], error: 'CONTROL_DIR not set' });

  // Read ledger files for needs_human status
  const ledgerDir = path.join(controlDir, 'ledger');
  const blockedTasks = [];
  for (const file of await readdir(ledgerDir)) {
    const content = await readFile(path.join(ledgerDir, file), 'utf-8');
    const entry = parseTaskLedger(content);  // key:value frontmatter parser
    if (entry.status === 'needs_human') {
      const specPath = path.join(controlDir, 'tasks', `${entry.task_id}.md`);
      const spec = await readFile(specPath, 'utf-8').catch(() => '');
      blockedTasks.push({ ...entry, spec });
    }
  }

  // Parse mailbox for per-task agent notes
  const mailbox = await readFile(path.join(controlDir, 'mailboxes', 'tshepo.md'), 'utf-8').catch(() => '');
  const notes = parseMailboxNotes(mailbox);  // returns Map<taskId, string>

  return c.json({ items: blockedTasks.map(t => ({ ...t, agentNote: notes.get(t.task_id) ?? '' })) });
});
```

**How the console unblocks a task (`POST /api/unblock/:taskId`):**
```typescript
app.post('/api/unblock/:taskId', async (c) => {
  const { taskId } = c.req.param();
  const { reason, agentName } = await c.req.json();
  if (!reason?.trim()) return c.json({ error: 'reason required' }, 400);

  const controlDir = process.env.CONTROL_DIR;
  const kernelPath = path.join(controlDir, 'kernel', 'task');

  // Run kernel/task unblock — this is a git commit in the control repo
  const result = await Bun.spawn([kernelPath, 'unblock', taskId, '--agent', 'tshepo', '--reason', reason]);
  const ok = (await result.exited) === 0;

  if (ok && agentName) {
    const knownAgents = getKnownAgentsFromFleetConf();  // validated at startup
    if (knownAgents.has(agentName)) {
      const mailboxPath = path.join(controlDir, 'mailboxes', `${agentName}.md`);
      const msg = `\n## from: tshepo | ${new Date().toISOString()} | re: ${taskId}\n${reason}\n`;
      await appendFile(mailboxPath, msg);
      await gitCommitAndPush(controlDir, [`mailboxes/${agentName}.md`], `mailbox: tshepo → ${agentName} re: ${taskId}`);
    }
  }
  return c.json({ ok });
});
```

**agentName path traversal protection:** `agentName` from the URL is validated against the Set of known agents read from `fleet.conf` at server startup. Unknown names return 400. This prevents path traversal attacks through the mailbox write path.

**What "Send back to agent" does (end-to-end):**
1. Human reads the blocked task card in the console (task title, agent's note, relevant ACs from the spec)
2. Human types their decision in the textarea and clicks "Send back to agent →"
3. Console POSTs `{ reason, agentName }` to `/api/unblock/:taskId`
4. Server runs `kernel/task unblock <id> --agent tshepo --reason "<decision>"` — this commits to the control repo, flipping `status: needs_human → open`
5. Server appends the decision to `mailboxes/<agent-name>.md` — the agent will read this on its next session start and see the resolution
6. The task is now eligible for any agent with the matching domain to pick up (or the original agent, when it wakes and finds it in the eligible pool)
7. If the supervisor's wake system is running, the original agent receives a wake signal within 1–5 seconds and starts a new session picking up the now-open task

**Mailbox write-back is a UX enhancement, not required for correctness.** The kernel's `task unblock` alone is sufficient to make the task eligible. The mailbox write-back gives the agent the human's decision as context when it picks up the task, so it doesn't have to re-derive what to do.

**Git push retry for mailbox write:** On push rejection, `git pull --rebase` then push again. One retry is sufficient for a 4-agent fleet:

```typescript
async function gitCommitAndPush(controlDir: string, files: string[], message: string) {
  const run = (args: string[]) => Bun.spawn(['git', '-C', controlDir, ...args]);
  await run(['add', ...files]).exited;
  await run(['commit', '-m', message]).exited;
  const push1 = await run(['push']).exited;
  if (push1 !== 0) {
    await run(['pull', '--rebase']).exited;
    await run(['push']).exited;  // second failure propagates to caller
  }
}
```

**Task ledger format** (`.task` files are plain key:value, one pair per line):
```
task_id: FEAT-011
status: needs_human
agent: agent-fe
role: feature
spec: tasks/FEAT-011.md
failure_count: 1
blocked_by:
lease_expires:
updated_at: 2026-06-19T09:28:00Z
```

**The console does NOT write to the task spec file** (`tasks/<TASK-ID>.md`). The spec is the agent's artifact; the console only reads it to surface context to the human.

**Failure count context for the UI:** If `failure_count >= 2`, the task has been attempted and blocked more than once. Show amber warning: "This task has been blocked twice — the AC may need rewriting."

## v1 Addition — Fleet Control, Work Pipeline, Stuck Detection

_Added 2026-06-20. Extends the v1 console with three new capabilities the v0 triage page does not cover._

### Why this is v1, not v0

v0 closes the synchronous loop (approve / unblock). v1 closes the **operational loop** — the things the human currently has to drop into a terminal to do:

1. **Fleet control** — start/stop/restart agents without opening a terminal (`fleet.sh start|stop|restart|install|uninstall`)
2. **Work pipeline** — see *what* each agent is doing right now and *what's queued behind it* (the kernel ledger surfaced visually)
3. **Stuck detection** — surface agents that have silently wedged (no events in N minutes, looping on the same tool, repeated failures on the same task) before the human has to notice

All three read from data the supervisor already produces. The only new server-side surface is a small set of POST endpoints that shell out to `fleet.sh`.

### 1. Fleet Control UI

**What it does.** A control strip per agent in the Fleet tab with Start, Stop, Restart buttons. A fleet-wide bar at the top for Start All / Stop All / Restart All / Install as service / Uninstall service. Every action shells out to the existing `fleet.sh` command — no new control plane.

```
│  agent-fe       bg         ⊗ needs-human  parked 2h 14m   [■][↻][logs]   │
│  agent-qa       service    ● working      Fetch · 4s ago  [■][↻][logs]   │
│  agent-doc      —          ○ stopped      —               [▶][logs]      │

```

**Endpoints (new, all POST, all 127.0.0.1-only, all CSRF-protected via same-origin check):**

| Endpoint | Shells to | Response |
|---|---|---|
| `POST /api/fleet/start` (optional `?agent=<name>`) | `fleet.sh start` | `{ ok, output }` — stdout/stderr captured |
| `POST /api/fleet/stop` (optional `?agent=<name>`) | `fleet.sh stop` | `{ ok, output }` |
| `POST /api/fleet/restart` (optional `?agent=<name>`) | `fleet.sh restart` | `{ ok, output }` |
| `POST /api/fleet/install` | `fleet.sh install` | `{ ok, output }` |
| `POST /api/fleet/uninstall` | `fleet.sh uninstall` | `{ ok, output }` |

**Per-agent scoping.** `fleet.sh` today operates on the entire fleet. The `?agent=<name>` mode is a console-level shim: it temporarily writes a single-line `fleet.conf` to a tmp file and runs `FLEET_CONF=<tmp> fleet.sh <cmd>`. Agent name validated against the in-memory Set (same as `/api/unblock`).

**Safety rails (mandatory before shipping):**
- **Confirm modal on Stop All / Restart All / Uninstall** — typed confirmation ("type STOP to confirm"). Single-agent stop is one-click; bulk is two-step.
- **Disable buttons during in-flight action** — the SSE `fleet-update` event clears the disabled state when state transition is observed.
- **Output panel** — every action opens a collapsible output panel under the row showing the last 20 lines of `fleet.sh` stdout/stderr. Errors render in red. Closes after 5s on success.
- **Localhost-only enforcement** — the same `127.0.0.1` bind from v0. Plus: every mutating endpoint checks `Origin === http://127.0.0.1:7842 || http://localhost:7842` and rejects otherwise. Cross-site can't trigger fleet shutdown.

**State source for buttons.** The Fleet table already polls `/api/fleet` (which parses `fleet.sh status` output and the presence files). The buttons read state from that same response — no separate state machine. After POST, the next SSE `fleet-update` event swaps the button set (`[▶][logs]` → `[■][↻][logs]`).

### 2. Work Pipeline View

**What it does.** Surfaces the kernel ledger as a visual pipeline so the human can answer "what is the team working on, and what's queued" without running `kernel/task ls`.

Three columns, one row per task, sourced from `$CONTROL_DIR/ledger/*.task`:

```

└──────────────────────────────────────────────────────────────────────────┘
```

**Column rules (status filter):**
- **In Flight** — ledger `status` ∈ `claimed`, `in_progress`. Sorted by claim time desc. Card shows agent, elapsed since claim, last tool name, last summary (from `live.json`).
- **Queued** — ledger `status` ∈ `open`, `ready`. Sorted by `priority` desc then `created_at` asc. Card shows domain, blocked_by chain if any, position in queue.
- **Blocked** — ledger `status` ∈ `needs_human`, `awaiting_info`, `failed`. Sorted by `updated_at` desc. Cards link to the existing Attention queue (for `needs_human`) or to a new agent-to-agent Q&A view (for `awaiting_info`, deferred to v2).

**Endpoint:** `GET /api/pipeline` — reads all ledger files, groups by column, attaches `last_tool`/`last_summary` from each agent's `live.json`. Cached for 2s (the ledger isn't a hot path). SSE event `pipeline-update` is fired when fs.watch sees any `ledger/*.task` change.

**Card click** opens a side drawer with the full task spec (read-only — same content as the Attention queue card). The drawer is shared infra with the Attention queue, not duplicated.

**Filter bar:** `[all] [feature] [qa] [doc]` chips at the top of the pipeline. Persists to localStorage. No URL state for v1 — single-user local app.

**What this is NOT:** This is not a kanban board you drag cards across. The pipeline is read-only — agents claim work themselves. The human's only mutations are via the Attention queue (unblock) or fleet control (stop the agent working on it).

### 3. Stuck Detection

**What "stuck" means.** Three orthogonal signals. Any one triggers the stuck badge; all three are surfaced separately so the human can diagnose.

| Signal | Definition | Source |
|---|---|---|
| **Silent** | `live.json.session_started_at` set, but no new event in `live-events.jsonl` for ≥ 5 minutes | `fs.stat` on live-events.jsonl + parse last line |
| **Looping** | Same `tool + summary[0..40]` appears ≥ 5 times in the last 20 events with no `task_claimed` between them | parse tail of live-events.jsonl |
| **Failing** | Same task has `failure_count ≥ 3` and is back in `status: open` (agent keeps re-claiming and failing) | ledger + event history |

**Where it surfaces:**

1. **Fleet table** — replace the state cell with `⚠ STUCK (silent · 7m)` in red when any signal fires. Hover shows all three signals' state.
2. **Pipeline view** — In Flight cards get a red ⚠ overlay and the "stuck since" timer.
3. **New "Stuck" alert at top of console** — slot above the Attention queue:

```
┌─ ⚠ STUCK ───────────────────────────────────────────────────────────────┐
│  agent-be has gone silent · last event 7m 22s ago · task FEAT-012        │
│  [View logs] [Restart agent] [Force-fail task]                           │
└──────────────────────────────────────────────────────────────────────────┘
```

**Endpoint:** `GET /api/stuck` — returns `{ items: [{ agent, signal, since, task, last_event }] }`. Re-evaluated on every SSE `tool_call` and `fleet-update` event. Pushed to clients via SSE event `stuck`.

**Actions in the stuck alert:**
- **View logs** — opens the same per-agent logs drawer as the Fleet tab `[logs]` button (tails `~/agents/<name>/logs/stdout.log` via SSE)
- **Restart agent** — POSTs `/api/fleet/restart?agent=<name>` with one-step typed confirm (typed: agent name)
- **Force-fail task** — POSTs new `/api/task/fail/:taskId` which shells out to `kernel/task fail <id> --agent tshepo --reason "human force-failed: stuck detection"`. Confirm modal required.

**Threshold tuning.** All thresholds (5min silent, 5x loop, 3 failures) are constants in `server.ts` for v1. After two weeks of live use, audit which threshold fired truly vs. falsely and move to a `console.config.json`.

**False-positive guardrail.** If `live.json.state` is `idle` (agent in `idle_wait`, by design), suppress ALL stuck signals for that agent. Silence is expected when descheduled.

## v1.1 Addition — Workspace Switcher

_Added 2026-06-20. Extends v1 with multi-project support so one console can direct multiple parallel engagements without re-launching._

### Why this exists

Today the supervisor assumes one fleet per workstation: a single `supervisor/fleet.conf`, a single global `~/agents/<name>/` namespace, a single set of launchd labels. The console inherits the same assumption. The moment you take on a second engagement, you either run a second console on a second port (and remember which tab is which), or you tear down and re-provision the first fleet.

A workspace is **one project = one control repo + N code repos + 1 doc repo + N agents**, fully isolated from every other workspace on the same machine. The switcher in the console is what lets one director triage across all of them from one tab.

This depends on a supervisor refactor (per-workspace `fleet.conf`, per-workspace agent homes, `--workspace` flag on `fleet.sh` and `run-agent.sh`, workspace-scoped launchd labels). That refactor is summarized in **Supervisor Workspace Refactor** below — it must land before the console switcher is wired up.

### What the human sees

**Switcher placement.** Top-left of the console header, left of every tab. Always visible. Color dot inherited from the workspace's `color` field. Clicking opens a popover:

```
┌─ ECOBA Engagement ▾ ────────────────────────────────────────────────────┐
│                                                                          │
│  ● ECOBA Engagement              ✓ active   4 agents · 2 attention      │
│  ● ACME Migration                            3 agents · 0 attention      │
│  ○ Sandbox (paused)                          1 agent  · 1 stuck         │
│                                                                          │
│  ─────────────────────────────────────────────────────────────────────  │
│  + Add workspace…                                                        │
│  ⚙ Manage workspaces                                                     │
└──────────────────────────────────────────────────────────────────────────┘
```

Each row shows: color dot, name, attention count (sum of approval queue + attention queue + stuck signals across that workspace). The active workspace has a checkmark. Workspaces with `paused: true` show a hollow dot.

**Cross-workspace rollup in the tab title.** Today the tab title is `(N) Fleet Console` where N = pending count of the active workspace. With workspaces, the format becomes:

```
(2 · ECOBA) Fleet Console            ← 2 pending in active workspace
(2+3) Fleet Console                  ← 2 in active + 3 across other workspaces
```

The `+3` is the sum across all non-active workspaces. Clicking the tab is the only way to see *which* workspace has the extra pending items — the switcher attention counts answer that on first click.

**Workspace badge on every card.** Every card (approval, attention, stuck, pipeline) carries a tiny workspace pill in the top-left meta row matching the workspace color. This is what lets the human cross-workspace if they ever flip a setting to "show all workspaces at once" (deferred — see open questions).

**Empty workspace.** A fresh-created workspace with no agents running shows the same "All clear — agents are running" page as v0, but the header reads `ECOBA Engagement · no agents running` and the primary CTA is `[Start fleet]` (calls `POST /api/fleet/start?workspace=ecoba`).

### Add workspace flow

Triggered by `+ Add workspace…` in the switcher. Modal, three steps. Wraps the existing `/setup-fleet` skill — the skill's logic is reused server-side, just collecting form input instead of asking interactive questions:

```
┌─ Add workspace ──────────────────────────────────────────────────────────┐
│  ① Identity                                                              │
│     Name         [ECOBA Engagement                                  ]    │
│     Slug         ecoba                  (auto · used in paths)           │
│     Color        ● amber  ● blue  ● green  ● purple  ● red               │
│                                                                          │
│  ② Repositories                                                          │
│     Control repo [git@github.com:seab-group/ecoba-control.git       ]    │
│                  ✓ valid · agents will share this for tasks & mailboxes  │
│     Doc repo     [git@github.com:seab-group/ecoba-docs.git          ]    │
│                                                                          │
│     Code repos   [git@github.com:seab-group/ecoba-api.git    ] be  ✕    │
│                  [git@github.com:seab-group/ecoba-web.git    ] fe  ✕    │
│                  [+ add code repo]                                       │
│                                                                          │
│  ③ Agents                                                                │
│     ☑ agent-be    · feature · be    · claude-sonnet-4-6                 │
│     ☑ agent-fe    · feature · fe    · claude-sonnet-4-6                 │
│     ☑ agent-qa    · qa      · qa    · claude-haiku-4-5                  │
│     ☑ agent-doc   · doc     · doc   · claude-haiku-4-5                  │
│       [+ add agent]                                                      │
│                                                                          │
│     ☐ Install as system services (auto-start at login)                  │
│                                                                          │
│  [ Cancel ]                              [ Create workspace → ]          │
└──────────────────────────────────────────────────────────────────────────┘
```

**On submit, server.ts does (in order):**

1. Write workspace entry to `~/.gstack-console/workspaces.json`
2. Create `~/agents/<slug>/` directory tree
3. Clone control repo to `~/agents/<slug>/console-checkout/` (the console's read-only checkout, not an agent's)
4. For each agent: create `~/agents/<slug>/<agent>/` and write the per-agent `config` file (`CONTROL_REPO_URL`, `WORK_REPO_URL` matched by domain, `DOC_REPO_URL`, `AGENT_DOMAIN`, `READ_REPOS` inferred as the other code repos, `SECRET_PREFIX=<slug>` for QA)
5. Write `~/agents/<slug>/fleet.conf` with the selected agent list
6. If "Install as system services" checked: invoke `fleet.sh --workspace=<slug> install`
7. Return the new workspace; UI switches to it

**Validation:**
- Slug must be lowercase `[a-z0-9-]+`, not already in use
- Each repo URL must be a parseable git URL (no live reachability check at create-time — agents will fail loud if URL is wrong, and that's surfaced via the stuck alert)
- At least one code repo required if any feature agent is selected
- At least one agent required

Reachability is **not** verified server-side because it requires the user's git credentials and could hang. The first time an agent runs it will fail-fast on bad URLs and the failure surfaces in the same Stuck alert and per-agent log drawer the human already uses.

### Manage workspaces flow

Triggered by `⚙ Manage workspaces`. Full-page tab, replaces the active workspace content while open. Lists all workspaces with row-level actions:

| Workspace | Repos | Agents | Status | Actions |
|---|---|---|---|---|
| ECOBA Engagement | 1 ctrl + 2 code + 1 doc | 4 (4 running) | active | [Open] [Edit] [Pause] [Delete] |
| ACME Migration | 1 ctrl + 1 code + 1 doc | 3 (3 running) | — | [Open] [Edit] [Pause] [Delete] |
| Sandbox | 1 ctrl + 0 code + 0 doc | 1 (stopped) | paused | [Open] [Edit] [Resume] [Delete] |

**Pause** = stop all agents, hide from the switcher attention rollup, but leave the workspace registered and its `~/agents/<slug>/` intact. Re-launch with **Resume**.

**Delete** = two-step typed confirm (typed: workspace name). Server runs `fleet.sh --workspace=<slug> stop`, then `fleet.sh --workspace=<slug> uninstall`, then `rm -rf ~/agents/<slug>/`, then removes the registry entry. The repos themselves are **not** deleted from GitHub — only the local checkouts. The confirm modal says exactly this.

**Edit** = lightweight: rename, change color, add/remove agents (which adds/removes per-agent directories and rewrites `fleet.conf`). Changing repo URLs requires Delete + recreate — surfaced as a warning in the edit modal.

### Server architecture changes

**Registry** at `~/.gstack-console/workspaces.json`:
```json
{
  "active": "ecoba",
  "workspaces": [
    {
      "id": "ecoba",
      "name": "ECOBA Engagement",
      "color": "amber",
      "agents_root": "~/agents/ecoba",
      "fleet_conf": "~/agents/ecoba/fleet.conf",
      "control_repo_url": "git@github.com:seab-group/ecoba-control.git",
      "console_checkout": "~/agents/ecoba/console-checkout",
      "doc_repo_url": "git@github.com:seab-group/ecoba-docs.git",
      "code_repos": [
        { "domain": "be", "url": "git@github.com:seab-group/ecoba-api.git" },
        { "domain": "fe", "url": "git@github.com:seab-group/ecoba-web.git" }
      ],
      "paused": false,
      "created_at": "2026-06-20T09:30:00Z"
    }
  ]
}
```

**In-memory state.** server.ts loads `workspaces.json` at startup and keeps a `Map<workspaceId, WorkspaceContext>` where each context holds: known-agent Set (read from that workspace's `fleet.conf`), the `fs.watch` handles for that workspace's agent log dirs, and the cached `CONTROL_DIR` (=`agents_root + '/console-checkout'`).

**Every existing v0/v1 endpoint gains a `workspace` parameter.** Query string for GETs (`?workspace=ecoba`), JSON body for POSTs (`{ "workspace": "ecoba", ... }`). If omitted, the server uses `active`. If the value isn't in the registry, the server returns 404 — no fallback to a default workspace, because silently routing a fleet command to the wrong project is the worst possible failure mode.

**SSE events carry `workspace_id`.** The single `/api/events` stream multiplexes across all workspaces. Every event payload gets a `workspace_id` field. The client subscribes once and filters in JS by the active workspace (and the cross-workspace rollup uses the unfiltered count).

**New endpoints:**

| Endpoint | Method | Purpose |
|---|---|---|
| `/api/workspaces` | GET | list all workspaces with attention counts |
| `/api/workspaces` | POST | create workspace (Add modal submit) |
| `/api/workspaces/:id` | PATCH | edit workspace (rename, color, agent list) |
| `/api/workspaces/:id` | DELETE | delete workspace (two-step confirm enforced client-side) |
| `/api/workspaces/:id/pause` | POST | stop agents + hide from rollup |
| `/api/workspaces/:id/resume` | POST | reverse of pause |
| `/api/workspaces/active` | PUT | switch active workspace (writes registry, returns full state) |

**Localhost + same-origin enforcement** carries over from v0/v1. The DELETE endpoint additionally requires the workspace's name in the request body (the typed-confirm value), server-side validated against the registry entry — defense in depth against an accidentally-triggered destructive action.

### Supervisor Workspace Refactor (prerequisite)

The console switcher does not work until the supervisor knows about workspaces. Summary of the supervisor-side changes — the full breakdown was given in the prior conversation:

| File | Change |
|---|---|
| `supervisor/run-agent.sh` | Accept `--workspace <slug>` flag → `AGENT_HOME="$HOME/agents/$WORKSPACE/$AGENT_NAME"`. Default to flat layout if flag absent (back-compat). |
| `supervisor/fleet.sh` | Accept `--workspace <slug>` → reads `~/agents/<slug>/fleet.conf`. PID dir becomes `~/agents/<slug>/.pids/`. Every subcommand routes through this scoping. |
| `supervisor/install.sh` | launchd label becomes `com.cstack.agent.<slug>.<name>`; plist `ProgramArguments` passes `--workspace <slug>`. |
| Per-agent `config` schema | Add `DOC_REPO_URL` alongside `CONTROL_REPO_URL` + `WORK_REPO_URL`. |
| `/setup-fleet` skill | Read `--workspace=<slug>` argument; write to `~/agents/<slug>/...` not `~/agents/...`. |
| Console: migration on startup | If `workspaces.json` is absent AND `~/agents/*/config` files exist in the flat layout, register one workspace called `default` pointing at the flat layout. Zero-touch migration. |

Nothing in the agent-to-agent protocol changes. Mailboxes, presence, wake signals all live inside `$CONTROL_DIR` which is already per-agent.

## v2 — React Cockpit (Modern, Human-Centered Reimagining)

_Added 2026-06-20. A re-architecture, not an addition. v0/v1/v1.1 earned the right to invest in a real component layer by proving the UX premise. v2 keeps the entire server side intact and replaces the htmx+vanilla frontend with a React app built around the question: how should humans actually work with AI colleagues in 2030?_

### Why react now (and not before)

The v0 "no build step" constraint was correct. We shipped, validated the loops, learned what the human actually needs. The cost of staying on htmx+vanilla now is the cost of every new interaction state being re-invented from scratch — animations, focus rings, drawer infra, keyboard shortcuts, multiplayer cursors, voice input, accessibility. Each one is a 2-day side-quest in vanilla. Each one is a 2-hour drop-in from a component library.

v2 is the bet that we've earned a real frontend. The server stays exactly as it is — Bun + Hono, SSE, the same endpoints. The browser just gets a real UI layer.

### Human-centered design principles for human-agent collaboration

The console is one of the first interfaces where humans direct an AI workforce, not a tool. The principles below are what we believe a 2030 director's surface needs. Every component decision in v2 traces back to one of these.

1. **Agents are colleagues, not commands.** Each agent has a face (generated avatar), a consistent color, a name, an optional voice. Presence is felt — when agent-be wakes up, you see her status dot warm from grey to green with a soft breath animation, not a blinking red badge.
2. **Conversations, not queues.** The unit of interaction is a thread, not a one-shot decision. When agent-fe asks "AC2 is ambiguous" you don't fill a textarea — you reply in a thread. Future messages on the same task append. The director can scroll back through every conversation they've ever had with that agent on that task.
3. **Trust grows over time.** The director can set "auto-approve `git commit` for agent-be after 30 successful approvals." The console tracks the counter; once met, future commits flow without intervention, with an "auto-approved (trust)" badge in the activity feed. The director can revoke trust per-agent-per-action with one click.
4. **Calm, not loud.** Default state is quiet. Ambient awareness lives in the periphery — color dots on agent avatars, subtle breath animations, a single number in the macOS dock. Escalations are earned: a stuck agent doesn't blink, it grows a halo. A high-risk approval doesn't ring, it gently nudges the page.
5. **Replay everything.** Every action carries a context scrubber. Click any agent message and the right pane shows: what the agent was doing 60 seconds before, what tool calls led to this, what changed in the repo. Like Linear's activity stream meets Loom's scrubber.
6. **One mind, many hands.** The director is the singular intelligence; the agents amplify their reach. The cmd-K palette is the universal input — typing "unblock feat-011 with reason: redirect to login" routes to the right action regardless of which screen you're on.
7. **Voice + keyboard first, mouse second.** Everything has a keyboard shortcut. Voice input on every textarea (Whisper API, local-first when MLX is ready). Designed for hands-free flow — fold laundry while triaging.
8. **Multiplayer when you want it.** Optional second human joins via a shared URL (Tailscale-gated). See each other's cursors, see who's about to approve what, comment on each other's decisions. Pair-debugging an agent fleet.
9. **Honest uncertainty.** When the AI co-pilot is unsure of a draft decision, it says so — not "Here is your answer" but "I'm 60% confident; here's what I'd ask the agent to clarify first."
10. **No dark patterns.** Confirmations are not nags. Notifications are opt-in. The console never plays a sound the user didn't choose. Auto-approval defaults are conservative; the human always opts in.

### New primitives that did not exist in v0/v1

These are net-new in v2 and embody the human-centered principles:

**Trust ledger.** A per-(agent, action-pattern) counter. Persisted to `~/.gstack-console/trust.json`. Schema:
```json
{
  "rules": [
    {
      "agent": "agent-be",
      "action_pattern": "^git commit -m",
      "successful_approvals": 23,
      "target": 30,
      "auto_approved_count": 0,
      "status": "learning",
      "created_at": "...",
      "revoked": false
    }
  ]
}
```
Once `successful_approvals >= target`, future matches auto-approve and emit an `auto_approved` event with `via: trust_rule:<id>`. The director sees the trust badge on every auto-approved action and can revoke with one click.

**Replay scrubber.** Every task's right-pane has a horizontal timeline. Hovering a point shows what the agent was doing then. Clicking jumps to that state. Surfaces the `live-events.jsonl` data the supervisor already produces; React turns it into a Loom-style scrubber.

**Cmd-K with natural language routing.** The command palette accepts free text. Server-side, a tiny Claude call (~$0.001) classifies the intent and returns a structured command. The user sees their typed text turn into an actionable suggestion: "→ Approve git push from agent-be (request_id: 8c4f)".

**Conversation thread per task.** Replaces the v0 textarea. Every message — agent question, human reply, agent acknowledgment, agent follow-up — is a turn in a thread persisted to `$CONTROL_DIR/conversations/<task-id>.jsonl`. The agent reads the thread on each session start so context is preserved across blocks.

**Multiplayer presence.** Optional. When enabled, the console exposes a Tailscale-gated read-only port; a second human visits `https://<machine>.<tailnet>/?invited=<token>` and sees the same surface with the director's cursor visible (Excalidraw-style live cursors via Yjs over the existing SSE channel). Pairing on agent decisions is one of the most-requested patterns from cstack interviews.

**Voice everywhere.** Every textarea has a `<VoiceInput />` mic button. Web Speech API in-browser for the common case (free, local). Whisper fallback for accuracy. Voice transcripts go through the same path as typed input, so AI draft, cmd-K, and conversation reply all accept voice equally.

**Sound design (calm).** Six total sounds, all under 200ms, all opt-in:
- Soft chime when an agent finishes a task
- Gentle pluck when an approval lands
- Warm hum when an agent goes stuck
- Two-tone "done" when the queue is fully clear
- Cmd-K open/close (subtle)
- Voice input start/stop (subtle)

Each sound has an off toggle; user can replace with their own file. Default is silent — sounds are opt-in.

### Migration path from v0/v1/v1.1

The server (`supervisor/console/server.ts`, all endpoints, SSE, file watchers) stays unchanged. Only the browser bundle changes. Hosting:

1. **Add `supervisor/console/web/`** — Vite project with `src/`, `index.html`, `vite.config.ts`. `package.json` workspace shares with the server.
2. **Server serves `web/dist/`** in production, proxies to `localhost:5173` in dev (Vite dev server). New env var `CONSOLE_DEV=1` triggers proxy mode.
3. **Endpoints are unchanged.** TanStack Query subscribes to the same `/api/queue`, `/api/attention`, `/api/pipeline`, `/api/stuck`. SSE `/api/events` becomes a TanStack Query "live query" via a tiny wrapper.
4. **Old `index.html` retained as `legacy.html`** for one release cycle; a banner on it links to the v2 UI. Removed when v2 hits parity.
5. **`bun run console`** is the dev command: starts Bun server + Vite together via `concurrently`. `bun run console:build` produces `web/dist/` for prod.

No data migration. No endpoint migration. Just a frontend swap.

### Performance budgets

| Metric | Budget | Why |
|---|---|---|
| Initial load (cold) | < 800ms TTI on M1 Mac | Local app — anything slower feels broken. |
| Bundle size (gzip) | < 200KB JS, < 30KB CSS | shadcn + React + TanStack ≈ 130KB; rest is our code. |
| SSE event → UI update | < 100ms | The whole point of live; anything slower defeats v0's gains. |
| Approval click → server ack | < 200ms | Optimistic update via TanStack mutation; rollback on failure. |
| Cmd-K open | < 50ms | Pre-mounted, hidden via CSS `transform`. |
| Voice input → transcript | < 1s for browser, < 3s for Whisper | Web Speech is faster; Whisper is the fallback. |

Enforce via Lighthouse CI in the build step. Bundle size regressions fail CI.

### Accessibility (non-negotiable, not v2.x)

- **WCAG 2.2 AA minimum.** Radix gives us this for free on every interactive primitive.
- **Keyboard-only operation 100%.** Every action reachable by Tab + Enter or by cmd-K. No mouse-only paths.
- **Screen reader.** Every card is an `<article>` with `aria-label`. Live regions for stuck alerts (`role="alert"`), composer state (`aria-live="polite"`).
- **Reduced motion.** All Framer Motion animations gated on `prefers-reduced-motion`. Sparklines become static bars; halos become solid borders.
- **High contrast.** A separate Tailwind theme variant (`data-theme="high-contrast"`) doubles contrast ratios. Toggle in settings.
- **Voice as accessibility tool.** Voice input is also voice output (optional): the system reads new conversation messages and stuck alerts via the Web Speech API.
